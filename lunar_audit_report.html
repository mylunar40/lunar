Architecture Analysis
✅ What's Done Well
  • Engine layer (core/engine/) — pure Dart, stateless, no Flutter imports. CycleEngine, InsightEngine, MoodEngine, MemoryExtractionEngine are all well-designed.
  • Repository pattern — correctly abstracts local cache from Firestore writes.
  • Pre-loading providers — init() is called before runApp(), so no loading flash.
  • ChangeNotifierProxyProvider used correctly to sync auth UID changes to ChatProvider and MemoryProvider.
  • Crashlytics + graceful Firebase failure — app falls back to demo mode when Firebase is unavailable.
CRITICAL  Dual Cycle State — Split Brain

UserProvider and LunarDataProvider both hold _lastPeriodDate and _cycleLength. Updating one does NOT update the other. Different screens read from different sources — they can show contradictory data.

UserProvider._lastPeriodDate    ← used in ai_insights, mood_tracking, home
LunarDataProvider._lastPeriodDate ← used in calendar, pregnancy, cycle_tracker
CRITICAL  Firestore Collection Mismatch

CycleRepository.pushToFirestore() writes to users/{uid}/cycles (subcollection).
FirestoreService.cycleStream() reads from top-level cycles collection.
Data written by the repository is never read back by the service.

HIGH  Duplicate Firestore Community Collections

Both community (legacy) and community_posts (current) exist. CommunityProvider reads from community_posts, but FirestoreService.communityStream() reads from community — creating a dead data path.

CRITICAL  No Named Routes / No go_router

Entire app uses imperative Navigator.push(). Deep linking (for push notifications) is impossible. When FCM notification arrives, there is no route to navigate to — despite firebase_messaging being in pubspec.yaml.

HIGH  Tab State Lost on Switch

_screens[_currentIndex] in MainNavigation builds a fresh widget tree on every tab switch. No IndexedStack — scroll position, loaded data, and animation state are lost every time the user changes tabs.

Performance Issues
HIGH IMPACT  7 Concurrent AnimationControllers on Home Screen

HomeDashboard runs 7 simultaneous repeating AnimationControllers (_floatController, _glowController, _orbitController, _particleController, _breatheCtrl, _pulseCtrl, _entryCtrl). ProfileScreen has 5, CalendarScreen has 6. These drive constant repaints.

HIGH  Canvas Particle Systems on Every Screen

Multiple screens maintain _StarParticle / _HeartParticle lists painted by CustomPainter every frame — 60+ repaints/second even when idle. No RepaintBoundary wrapping animated sections.

MEDIUM  CommunityProvider Starts Firestore Listener at App Start

Subscribes to communityFeedStream on construction (root level), not on first screen visit. Every user pays this cost even if they never open the Community tab.

Security Issues
🚨 CRITICAL  Authentication Bypass When Firebase Unavailable
// auth_provider.dart line ~30
bool get isAuthenticated =>
    !_firebaseAvailable || _firebaseUser != null;  // ← DANGEROUS

When Firebase fails to initialize, _firebaseAvailable = false so isAuthenticated returns true unconditionally. The app grants full access to ALL features. This is a deliberate demo-mode decision but is a critical security hole if Firestore is intermittently down.

CRITICAL  Firebase API Key in Source Control
// firebase_options.dart
static const FirebaseOptions android = FirebaseOptions(
  apiKey: 'AIzaSyCDOKtvoJhy_mMqafXA2mNs0Vhdc5q_EkQ',  // ← EXPOSED
  projectId: 'lunar-ai-8e8dc',
);

If this repo is ever made public, the project ID can be used for quota exhaustion attacks. Firebase rules must be tight to compensate.

CRITICAL  OpenAI API Key in SharedPreferences

The OpenAI key is stored via LocalCache.setString() (SharedPreferences). On rooted Android devices or via ADB backup, this key is extractable. Must use flutter_secure_storage (Android Keystore / iOS Keychain).

HIGH  Account Deletion Doesn't Clean Firestore Data

AuthService.deleteAccount() only deletes the Firebase Auth account. All user data in users, moods, journals, cycles, ai_memory collections remains permanently — violating GDPR data minimization.

HIGH  No Firestore Security Rules File in Repo

No firestore.rules file exists in the project. Rules are managed in Firebase Console only — invisible to code review and version control. Any team member or audit cannot see what data is protected.

MEDIUM  No Email Verification · No Rate Limiting · No Input Length Limits
  • Users can access all features immediately after signup without verifying email
  • No per-user rate limiting on OpenAI API calls — one user can exhaust the quota
  • createCommunityPost() sends content directly to Firestore with no max length check

Code Quality Issues
HIGH  Color Constants Duplicated in 15+ Files

Every screen redefines the same 8–10 color constants. ThemeData is set up in main.dart but never used — screens access colors from local const variables instead of Theme.of(context).

// Defined in main.dart AND in every single screen:
const Color _kBg     = Color(0xFF0A0118);   // 15+ copies
const Color _kPurple = Color(0xFFAB5CF2);  // 15+ copies
CRITICAL  JournalScreen is Non-Functional

journal_screen.dart stores entries in an in-memory List<Map>. Data is lost on hot restart. No Provider, no Firestore, no SharedPreferences. Uses default Material Scaffold (no dark theme). The full JournalEntry model and Firestore journalStream / saveJournal services already exist — and are completely unused.

CRITICAL  MoodAnalyticsScreen Has Hardcoded Fake Data
List<FlSpot> moodSpots = [
  FlSpot(0, 3), FlSpot(1, 4), FlSpot(2, 2),  // ← FAKE static data
  FlSpot(3, 5), FlSpot(4, 4), FlSpot(5, 3), FlSpot(6, 4),

Not connected to real mood entries from LunarDataProvider. Shows the same fake graph to every user.

HIGH  SleepScreen Has No Data Persistence

sleep_screen.dart is UI-only — no connection to LunarDataProvider.sleepLogs or any Firestore write. Sleep data entered by the user disappears on restart.

MEDIUM  Other Code Quality Issues

  • Duplicate file: both lib/user_provider.dart and lib/screen/user_provider.dart exist

Scalability Improvements
🗺️ S1 — Replace Navigator.push with go_router

Enables deep linking, push notification navigation, URL-based routing for web, and named routes. This is the single highest-value architectural change for long-term scalability.

🎨 S2 — Centralize Color System into LunarTheme

Create lib/theme/lunar_theme.dart. Replace 15+ local const Color _k* redefinitions. Screens should use Theme.of(context).colorScheme. Enables theme switching in the future.

🗑️ S3 — Remove UserProvider, Unify Under LunarDataProvider

UserProvider is a subset of LunarDataProvider. Eliminate dual state. Update the 5–6 screens still referencing the legacy provider.

📌 S4 — IndexedStack for Bottom Navigation

Preserves tab state on switch. Prevents rebuilds and Firestore listener re-subscriptions every time a tab is tapped.

🔒 S5 — flutter_secure_storage for API Keys

Replace SharedPreferences for the OpenAI API key. Uses Android Keystore / iOS Keychain. Cannot be extracted from rooted devices.

🏗️ S6 — Build Flavors (dev / staging / prod)

Replace isDevelopmentMode constant with proper Flutter build flavors and separate Firebase projects per environment.

🖼️ S7 — RepaintBoundary for Particle Animations

Wrap all CustomPaint particle painters and glow animations in RepaintBoundary to limit repaint scope to the animated widget only — not the entire screen.

💤 S8 — Lazy-initialize CommunityProvider

Subscribe to the Firestore feed only when CommunityScreen is first mounted, not at app startup. Reduces unnecessary network usage and Firestore reads.
